# Data Migration Pipeline: Base44 → Supabase

> Created 2026-04-02. Repeatable pipeline for moving all entity data.

---

## Pipeline Overview

```
Base44 Entities
      │
      ▼
  ┌─────────┐     ┌────────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
  │  Export  │ ──▶ │  Identity  │ ──▶ │Transform │ ──▶ │  Import  │ ──▶ │  Verify  │
  │ (Base44) │     │    Map     │     │(normalize│     │(Supabase)│     │ (counts  │
  │          │     │(Base44 ID →│     │ + remap) │     │          │     │  + FKs)  │
  └─────────┘     │ Supa UUID) │     └──────────┘     └──────────┘     └──────────┘
                  └────────────┘
```

**Key principle:** Source data is never modified or deleted. All scripts are read-only on Base44, write-only to Supabase.

## Quick Start

```bash
# 1. Start local Supabase
npx supabase start
# Note the service_role key from output

# 2. Set environment
export SUPABASE_SERVICE_ROLE_KEY=<key-from-step-1>
export SUPABASE_URL=http://127.0.0.1:54321

# 3. Run full pipeline
bash scripts/migration/run-migration.sh all

# Or run individual steps:
bash scripts/migration/run-migration.sh export
bash scripts/migration/run-migration.sh identity
bash scripts/migration/run-migration.sh transform
bash scripts/migration/run-migration.sh import
bash scripts/migration/run-migration.sh verify

# Dry run (validate without inserting):
bash scripts/migration/run-migration.sh dry-run
```

## Directory Structure

```
scripts/migration/
  config.js                          # Shared config: paths, entity→table mapping, Supabase client
  run-migration.sh                   # Orchestrator script (runs all steps)
  export/
    export-base44.js                 # Export all Base44 entities to JSON
    data/                            # Output: {EntityName}.json per entity
      _identity_map.json             # Generated by build-identity-map step
  transform/
    build-identity-map.js            # Base44 ID → Supabase UUID mapping
    transform.js                     # Normalize + remap all exported data
    data/                            # Output: {table_name}.json per table
  import/
    import-supabase.js               # Bulk upsert into Supabase tables
  verify/
    verify.js                        # Row counts + FK integrity checks
  logs/
    export-*.jsonl                   # Export timing/error log
    transform-failures-*.jsonl       # Records that failed validation
    transform-summary-*.jsonl        # Per-entity transform stats
    import-*.jsonl                   # Import batch results
    verify-failures-*.jsonl          # Verification failures
    identity-conflicts.json          # Users with duplicate emails or missing IDs
```

## Step-by-Step

### Step 1: Export (`export-base44.js`)

Exports all 31 Base44 entities to individual JSON files.

**Input:** Live Base44 connection (requires valid `VITE_BASE44_*` env vars and `VITE_USE_MOCKS=false`)

**Output:** One file per entity in `export/data/`:
```json
{
  "entity": "User",
  "exported_at": "2026-04-02T...",
  "count": 150,
  "records": [ { "id": "...", "email": "...", ... }, ... ]
}
```

**Note:** The export script uses the Base44 SDK's `.list()` method with service role access. If the SDK doesn't support Node.js execution, export manually from the Base44 dashboard and format as the JSON structure above.

### Step 2: Build Identity Map (`build-identity-map.js`)

Creates a mapping between Base44 user IDs and Supabase UUIDs.

**Input:** `export/data/User.json`

**Output:** `export/data/_identity_map.json`
```json
{
  "total_users": 150,
  "mapped": 148,
  "conflicts": 2,
  "byId": {
    "base44-user-id-123": "a1b2c3d4-e5f6-4789-abcd-ef0123456789"
  },
  "byEmail": {
    "john@example.com": "a1b2c3d4-e5f6-4789-abcd-ef0123456789"
  }
}
```

**ID strategy:**
- If Base44 ID is already UUID format → keep it
- Otherwise → generate deterministic UUID from SHA-256 hash of the Base44 ID

**Conflict handling:** Users with duplicate emails or missing IDs are logged to `logs/identity-conflicts.json` and excluded from the map.

### Step 3: Transform (`transform.js`)

Applies all data transformations to convert Base44 format to Supabase schema.

**Transformations applied:**

| # | Transformation | Affected Tables |
|---|---------------|----------------|
| 1 | Field renames: `created_date`→`created_at`, `updated_date`→`updated_at` | All tables |
| 2 | Soft-delete: `is_deleted` (bool) → `deleted_at` (timestamp) | direct_messages, direct_message_files, circle_messages, circle_files |
| 3 | ID resolution: Base44 user IDs → Supabase UUIDs | All tables with user_id, sender_user_id, etc. FKs |
| 4 | Email→UUID: `actor_email` → `actor_id` FK | audit_logs |
| 5 | Email→UUID: `email` → `user_id` FK | lawyer_applications |
| 6 | Email normalization: lowercase + trim | All email fields |
| 7 | Array normalization: JSON strings → PostgreSQL arrays | states_licensed, practice_areas, tags, etc. |
| 8 | Validation: required fields, enum values | All tables |

**Output:** One file per table in `transform/data/`:
```json
{
  "table": "profiles",
  "source_entity": "User",
  "phase": 1,
  "count": 148,
  "failed": 2,
  "records": [...]
}
```

**Failure handling:** Records that fail validation are logged to `logs/transform-failures-*.jsonl` with the original record and error details. They are excluded from the output file.

### Step 4: Import (`import-supabase.js`)

Bulk upserts transformed data into Supabase using the service_role key (bypasses RLS).

**Import order (5 phases, respects FK constraints):**

| Phase | Tables |
|-------|--------|
| 1 | profiles, lawyer_profiles |
| 2 | leads, blog_posts, content_posts, resources, mass_torts, popups, email_verification_otps |
| 3 | legal_circles, cases, lawyer_applications, activation_tokens, attorney_invitations, direct_message_threads |
| 4 | legal_circle_members, legal_circle_invitations, legal_circle_cases, direct_message_participants, direct_messages, circle_messages, circle_documents, circle_notifications, resource_events, popup_impressions, consent_logs, invitations, audit_logs |
| 5 | direct_message_files, circle_files, document_versions, document_signatures |

**Batch size:** 500 records per upsert call to avoid timeouts.

**Upsert behavior:** Uses `onConflict: 'id'` — existing records are updated, new records are inserted. Source data is never deleted.

**Options:**
```bash
# Import only phase 3
node scripts/migration/import/import-supabase.js --phase 3

# Import only a specific table
node scripts/migration/import/import-supabase.js --table profiles

# Dry run (validate without inserting)
node scripts/migration/import/import-supabase.js --dry-run
```

### Step 5: Verify (`verify.js`)

Checks migration integrity after import.

**Checks performed:**

| Check | What It Verifies |
|-------|-----------------|
| Row counts | Transform output count == Supabase table count |
| FK integrity | All FK references point to existing rows (no orphans) |
| Profile completeness | All profiles have email addresses |

**Exit code:** 0 if all checks pass, 1 if any fail.

## Re-running the Pipeline

The pipeline is designed to be run multiple times safely:

- **Export** overwrites previous export files (data is re-fetched fresh)
- **Identity map** is regenerated from the latest User export
- **Transform** overwrites previous transform files
- **Import** uses upsert — existing records updated, new records added
- **Verify** runs fresh checks against current Supabase state
- **Logs** are append-only (timestamped filenames)

To re-run from scratch:
```bash
rm -rf scripts/migration/export/data scripts/migration/transform/data
bash scripts/migration/run-migration.sh all
```

## Known Limitations

1. **Base44 export requires live connection** — the export script uses the Base44 SDK. If the SDK can't run in Node.js, export manually from the Base44 dashboard.

2. **Password hashes are not exportable** — all users will need to reset passwords after migration. See `docs/migration/auth-migration.md` for the password reset strategy.

3. **File URLs remain as-is** — uploaded files stay at their Base44 storage URLs. A separate file migration to Supabase Storage is recommended but not included in this pipeline.

4. **Real-time subscriptions** — the data pipeline migrates static data only. Real-time (Supabase Realtime) is configured by the schema migrations, not by data import.

5. **No rollback** — the pipeline doesn't support rolling back imported data. To start fresh: `npx supabase db reset` to wipe and re-apply migrations.
